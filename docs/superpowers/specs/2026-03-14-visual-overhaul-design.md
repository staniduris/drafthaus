# Visual Overhaul: Tailwind + AI-Driven Styling

## Problem

Drafthaus renders structurally correct but visually flat HTML. Design tokens (colors, fonts, spacing, radius, density) are stored but barely used. Fonts are declared but never loaded. The homepage shows raw data ("3.5" instead of "$3.50"), uses the first entity's name as the page title, and every site gets the same Stack > Grid > Card layout regardless of content type.

The target: sites generated by `drafthaus generate "cafe in Bratislava"` should look like premium WordPress/Squarespace themes — rich typography, visual hierarchy, alternating section backgrounds, proper spacing, formatted data.

## Approach

Replace the hand-written `dh-*` CSS system with Tailwind CSS embedded at build time. Primitives accept a `class` prop for Tailwind utility classes. The AI generates styling decisions (class strings) alongside content. The Go code provides smart defaults for non-AI paths (templates, manual init).

## Architecture

### Build pipeline

```
go generate ./...
  └─ runs: npx tailwindcss -i input.css -o embed/tailwind/dist.css
  └─ input.css includes: @tailwind base; @tailwind components; @tailwind utilities;
  └─ tailwind.config.js: safelist: [{pattern: /.*/}] — ships the FULL Tailwind CSS
  └─ embed/tailwind/dist.css is embedded via //go:embed into the binary (~300KB gzipped)
```

**Why ship full Tailwind:** AI-generated classes are unbounded at runtime — the build step cannot predict what the AI will generate. Purging would break AI-generated sites.

**Size tradeoff:** Full Tailwind v3 is ~4MB raw / ~300KB gzipped. In a `<style>` tag browsers parse the full CSS, which adds ~50-100ms on mobile. This is acceptable for a CMS that generates rich pages. If profiling shows this is a problem, we can later subset to commonly-used categories (layout, spacing, typography, colors, borders, effects, transforms — ~1.5MB raw) while safelisting the rest.

Build dependency: Node + npx (dev only). Runtime: zero dependencies (unchanged).

### Rendering changes

Every primitive gains a `class` prop. The primitive's Go code merges it with any structural classes:

```go
// Before
func sectionPrimitive(n *Node, ctx *RenderContext) (string, error) {
    return fmt.Sprintf("<section class=\"dh-section\">%s</section>", children), nil
}

// After
func sectionPrimitive(n *Node, ctx *RenderContext) (string, error) {
    cls := mergeClasses("dh-section", PropString(n, "class"))
    return fmt.Sprintf("<section class=\"%s\">%s</section>", cls, children), nil
}
```

The `dh-*` classes remain as structural defaults (flexbox layout, grid setup) but are minimal. All visual styling comes from Tailwind classes in the `class` prop.

**Security:** The `mergeClasses` helper sanitizes the `class` prop — only characters matching `[a-zA-Z0-9\-_:/[\]().% ]` are allowed. This prevents XSS via malformed class attributes from AI output.

### CSS output

`buildHTMLDoc` changes:

1. Emit Google Fonts `<link>` by reading `tokens.Fonts`, filtering system fonts, and building the Google Fonts URL
2. Emit the embedded Tailwind CSS in a `<style>` tag
3. Emit a small `<style>` block with CSS custom properties from tokens (`:root { --color-primary: #92400E; }`) so Tailwind's `bg-[var(--color-primary)]` works
4. Keep the minimal `dh-*` structural CSS (~30 lines for grid/flex defaults)

### AI prompt changes

The `generateSystemPrompt` in `internal/ai/generate.go` expands the SiteSpec schema to include view trees with Tailwind classes:

```json
{
  "site_name": "The Golden Bean",
  "colors": { ... },
  "fonts": { ... },
  "entity_types": [ ... ],
  "views": {
    "Homepage": {
      "type": "Stack",
      "children": [
        {
          "type": "Section",
          "props": {
            "class": "min-h-[80vh] flex items-center justify-center text-center bg-gradient-to-br from-stone-900 to-stone-700 text-white"
          },
          "children": [
            {"type": "Heading", "props": {"level": 1, "class": "text-6xl font-bold tracking-tight", "text": "..."}},
            {"type": "Text", "props": {"class": "text-lg text-white/70 max-w-lg mx-auto mt-6"}},
            {"type": "Action", "props": {"class": "mt-8 bg-amber-800 hover:bg-amber-700 px-8 py-4 rounded-xl uppercase tracking-widest", "label": "View Menu", "href": "/menu"}}
          ]
        },
        {
          "type": "Section",
          "props": {"class": "py-20 bg-amber-50"},
          "children": [...]
        }
      ]
    }
  }
}
```

The AI generates the full view tree with styling. The Go `ApplySiteSpec` function stores these views directly.

### Smart Go defaults (non-AI path)

For `drafthaus init --template cafe` (no AI), the template builders in `internal/cli/init.go` include hardcoded Tailwind classes that produce good-looking output:

- Hero: dark gradient, large centered heading, CTA button
- Menu grid: shadowed cards with category badge, price, description
- Sections alternate between white, surface, and dark backgrounds
- Footer: dark with muted text

### Font loading

New function in `internal/render/css.go`:

```go
func GenerateFontLink(tokens *draft.Tokens) string {
    // Collect unique font families from tokens.Fonts
    // Skip system fonts (system-ui, sans-serif, monospace, Georgia, etc.)
    // Build: <link href="https://fonts.googleapis.com/css2?family=Playfair+Display:ital,wght@0,400;0,700;0,800;1,400;1,700&family=Inter:wght@400;500;600;700&display=swap" rel="stylesheet">
}
```

**Portability rule:** `drafthaus serve` emits Google Fonts `<link>` (live sites have internet). For offline/demo scenarios, the CSS includes system font fallbacks (`font-family: 'Playfair Display', Georgia, serif`) so pages remain readable without network. `drafthaus export` downloads the font woff2 files into `_fonts/` and emits `@font-face` rules with local paths. If font download fails (network error, unknown font), export logs a warning and falls back to system fonts — it does not fail.

Called by `buildHTMLDoc` before the `<style>` block.

## Bug fixes (included in this work)

1. **Homepage title**: `RenderList` sets `bindCtx.Entity = entities[0]` which makes `extractTitle` return the first entity's name. Fix: when rendering homepage, use `tokens.SiteName` as the title instead.

2. **JSON-LD unresolved URL**: `buildJSONLD` replaces `:slug` but routes use `{slug}`. Fix: also replace `{slug}` pattern.

3. **RSS feed title**: Uses `chosen.Name` (entity type name like "MenuItem"). Fix: use `tokens.SiteName` with type name as fallback.

4. **Price formatting**: Already works in the `Price` primitive (formats with currency symbol). The issue is that Go view tree builders use `Text` with bind instead of `Price`. Fix: use `Price` primitive for currency fields in `buildListCardChildren`.

5. **site_name in Colors map**: This is a hack. Fix: add `SiteName string` field to `draft.Tokens` struct. Migrate existing .draft files by reading `Colors["site_name"]` into the new field on load, then deleting the key from the map. Update all code that reads `tokens.SiteName` to use `tokens.SiteName`.

## What this does NOT include

- Image generation (AI-generated photos for cards/hero) — separate spec
- Drafthaus Cloud hosting — separate spec
- New primitives — existing set is sufficient (RegisterAll registers 29 primitives as of v1.0)
- Admin UI changes — admin already works, visual output is the priority
- WASM plugin changes — plugins can use class prop same as built-in views

## Success criteria

1. `drafthaus init mysite --template cafe && drafthaus serve mysite.draft` produces a page that looks like the `/tmp/drafthaus-rich.html` mockup — rich hero, styled cards, proper typography, formatted prices
2. `drafthaus generate "coffee shop" --provider openai` produces a unique, visually rich site with AI-chosen Tailwind styling
3. `drafthaus export` produces static HTML with all CSS inlined (no CDN dependency)
4. Binary size stays under 25MB (currently 21MB, full Tailwind CSS adds ~300KB uncompressed)
5. All existing tests pass
6. Homepage title shows site name, not first entity name
7. Google Fonts load when serving; exported sites include local font files
8. Class prop values are sanitized against XSS
